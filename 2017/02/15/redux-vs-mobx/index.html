React state管理解决方案对比(Redux vs Mobx)

2017-02-15
  • redux,mobx

Redux 和 Mobx 是两个比较流行的 react state 管理解决方案，本文主要来对比下两个解决方案及其各自适合的使用场景。

不同之处

redux

  • 函数式编程

函数式变成已经变得非常流行，但是大多数开发者更熟悉的是面向对象编程模式，所以 redux 对于这些开发者来说具有一定的起步成本

  • 纯函数

利用函数式编程中的纯函数方式，纯函数接受输入参数，输出输出对象，且不依赖任何外部属性、对象；纯函数在接受固定的输入参数之后，一定会返回相同的输出值，并且不会产生任何 side effect

  • Immutable

redux 规定 state 应该是 immutable 的，每次对于 state 的修改都应该返回一个新 state 对象，而不是直接修改 state 对象

  • Normalized State

在《Redux最佳实践之关系型State对象管理》一文中介绍了 redux 推荐使用类似数据库的关系型数据结构来存储 state 数据

  • Side Effects

类似异步请求等无法满足纯函数解决方式的场景，可参考：

  • redux-thunk
  • redux-promise
  • redux-saga
  • redux-obserable

等解决方案

mobx

mobx 深受 Object-oriented Programming 和 Reactive Programming，mobx 可以通过类的方式声明一个数据对象，再通过 ES7 observable 的方式让对象属性具备可监听属性，一旦其值改变，就可以得知数据变更

由于 observable 的特性，mobx 对于数据的修改可以通过更直接的方式进行：

数据修改之后，mbox 借助 observable 的特性可以获取数据变更，再告知 react component 更新视图，类似 vue.js 的 MVVM 模式，所以在 mobx 中只存在 state => view 两层

不同点

性能

性能是 mobx 的一大特色，借助 mobx-react，mobx 可以做到render需要render的组件，只有相关的状态变更才会影响 view 的重新渲染

这里面的原理也比较粗暴，mobx 重写了 react 的 shouldComponentUpdate 方法，只有在 react component 关联的 observable 属性/对象变更之后会才执行重新渲染

mobx 不需要关注性能优化！这个是一个很屌的特性。。。

OOP vs Pure Function

两种完全不同的编程方式，redux 只允许使用 action 来触发 reducer 对于 state 的变更，而 reducer 是一个 Pure Function

mobx 可以直接调用对象的方法进行修改，或直接修改暴露出来的对象属性；新版本 mobx 也借鉴了 redux 的 action 的概念，当申明一个应用为严格模式的情况下，也只能通过 action 来触发 state 变更

个人小结

学习曲线

mobx 只引入了 observable 的概念，而 redux 则包含 action => reducer、函数式编程等概念，从前期学习成本上来说，mobx 无疑更适合入门

但是，mobx 利用 observable 封装了很多实现细节，而这些实现细节是我们无法轻松了解的；相反，redux 暴露出来的使用方式就是其内部原理，所以相对来说，两者的学习曲线一个是前面轻松、后面陡峭，另一个则是前面陡峭、后面轻松

开发效率

小型项目更适合使用 mobx，更熟悉的开发模式以及更少量的非业务代码；大型项目来说，两者的开发效率应该差别不大

规范 vs 自由

使用 redux 开发应用，由于限定了 action => reducer => state 的架构层级，对应的文件目录结构也已经规范如此；而 mobx 则只规范了 react component 的 observable 范式，其它完全没有规范

在大型的多人开发的项目中，更规范的开发范式可以减少沟通成本，也可以减少一些不必要的犯错机会
