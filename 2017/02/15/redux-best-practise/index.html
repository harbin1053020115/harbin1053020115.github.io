React & Redux 最佳实践

2017-02-15
  • react
  • redux

业界方案

react + react-router + redux + react-router-redux + redux-saga(可选)

关于异步处理

redux 可解决了同步方式的 Action <=> State 业务逻辑处理，异步处理业界也有相应的解决方案。比如：

  • redux-thunk
  • redux-promise

通过 action creator 的方式返回一个函数来封装异步操作以及后续的 action dispatch，可以解决问题，但是会影响 component 的纯粹性（dispatch 可能会分散到 component 中）

  • redux-saga

用于管理 action，处理异步逻辑。可测试、可 mock、声明式的指令

component 只需要 dispatch 一个 action，后续所有的操作（包括异步操作，dispatch 另一个 action）都由 saga 进行管理；同时，saga 利用 generator 来进行异步操作的同步方式 coding，让代码更可读性 & 可维护性更高

另外，引入 saga 之后还会同步引入一个 saga 层，让本身就并不简单的 redux 目录结构更加复杂，增加了开发复杂度

整体开发感受

概念太多，上手难度高

  • saga 理解成本高，书写复杂
  • 开发成本高，需要频繁在 reducer, saga, action 之间来回切换
  • 不方便组织业务模型，创建一个业务模型需要创建一堆文件
  • entry 文件书写复杂：除了 redux store 的创建，中间件的配置，路由的初始化，Provider 的 store 的绑定，saga 的初始化，还要处理 reducer, component, saga 的 HMR

支付宝解决方案

dva

基于以上开发问题，支付宝基于现有解决方案封装了 dva

两大特性：

  • 借鉴 choo API 简化 entry 文件书写
  • 借鉴 elm 引入 model 概念，用于把 reducer, initialState, action, saga 封装到一起

带来的好处：

  • 5 步 4 个接口完成单页应用的编码，不需要配 middleware，不需要初始化 saga runner，不需要 fork, watch saga，不需要创建 store，不需要写 createStore，然后和 Provider 绑定，等等。但却能拥有 redux + redux-saga + … 的所有功能
  • 封装 redux & saga 中的 reducer, action, saga 到一个 modal 文件，集中管理所有业务逻辑

待优化点

  • Enhance reducer 暂不支持
  • 动态的加载 modal 暂不支持
  • 脚手架优化点：
    • 多页面场景，支持添加页面
    • 符合数加平台这边的 util/request 封装
    • 接入云构建
